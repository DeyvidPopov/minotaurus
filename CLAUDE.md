# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo layout

Two-tree monorepo with no root `package.json`. Always `cd` into the right tree first.

- `backend/` — Express + TypeScript + Prisma/PostgreSQL. ESM (`NodeNext`), run via `tsx` (no separate build step in dev).
- `frontend/nextjs/` — Next.js 14 App Router, React 18, Tailwind, React Flow, Mermaid, Zustand. Talks to the backend over HTTP through `lib/api/`.
- `frontend/src/`, `frontend/tweaks-panel.jsx`, `frontend/index.html` — **legacy** Vite/JSX prototype, kept only as a design reference. Do not edit unless explicitly asked.
- `docs/` — original product spec (`01_..11_*.md`) plus `docs/AI_CONTEXT/` which holds the **living** state docs (`CURRENT_STATE.md`, `FEATURE_PROGRESS.md`, `KNOWN_LIMITATIONS.md`, `NEXT_STEPS.md`, `SESSION_HANDOFF.md`, `ARCHITECTURE_RULES.md`). Read those before guessing at intent — they are kept current per feature pass.
- `FRONTEND_RUNTIME_NOTES.md` — short-form local-runtime walkthrough (also covers the seeded demo flow).

## Common commands

### Backend (`cd backend`)

```
npm run dev              # tsx watch src/server.ts  (backend on :4000)
npm start                # tsx (no watcher)
npm run build            # prisma generate && tsc -p tsconfig.json  → dist/
npm run seed             # tsx scripts/seed-demo.ts — wipes + reseeds demo dataset
npm run test:api         # bash scripts/test-api.sh — full HTTP smoke pass (needs backend running)
npm run test:unit        # node --import tsx --test "src/**/*.test.ts" — pure-logic unit tests
npm run typecheck        # tsc -p tsconfig.json --noEmit — use THIS, not a bare `tsc`/`npx tsc`
npm run prisma:generate  # regenerate Prisma client
npm run prisma:migrate   # prisma migrate dev (interactive)
npm run prisma:reset     # guard-destructive.ts && prisma migrate reset --force (DESTRUCTIVE, guarded)
npm run prisma:studio    # Prisma Studio GUI
```

Two test layers: `scripts/test-api.sh` is the HTTP smoke pass, and `test:unit` runs Node's built-in test runner (`node:test` + `node:assert`, no Jest/Vitest) over `*.test.ts` files for pure-logic modules — currently the Export Engine V2 analysis engine, PDF SVG normalizer, AI proposal/review engines, JWT-secret config guard, async-error handler, and destructive-script guard. New pure engines should ship a colocated `*.test.ts`. To run a single endpoint, hit it with `curl` (patterns in `backend/API_TEST_EXAMPLES.md`).

**Always typecheck via `npm run typecheck`, never a bare `tsc`/`npx tsc`.** A global `tsc` may sit on `PATH` (seen as an old 3.9.7) and shadow the pinned local `typescript`; npm scripts prepend `node_modules/.bin`, so the script is the only PATH-order-independent entry point. (Same reason CI must call `npm run typecheck`.)

**Regenerating the Prisma client on Windows (DLL lock).** A running `npm run dev` (`tsx watch`) holds `node_modules/.prisma/client/query_engine-windows.dll.node`, so `prisma generate` fails with `EPERM … rename …tmp…` (leaving stray `.tmp` files) **and** the still-running server keeps the *old* client in memory — a newly added column then throws `Unknown argument \`<field>\`` at runtime even though `schema.prisma`, the migration, and the DB columns are all correct. A hot-reload won't save you: `tsx watch` only watches `src/`, not `node_modules`, and `lib/prisma.ts` caches the client on `globalThis`. Fix: **fully stop the backend** (kill the `tsx watch` process *and* its server child — and check for a second stale server still bound to `:4000`), then `npx prisma generate`, then restart. Verify with `npx prisma migrate status` (schema/DB) separately from the client regen — they fail independently.

### Frontend (`cd frontend/nextjs`)

```
npm run dev        # next dev    (frontend on :3000)
npm run build      # next build
npm run start      # next start
npm run lint       # next lint
npm run typecheck  # tsc --noEmit
```

**To verify a frontend change, use `npm run typecheck`, not `npm run build`.** `next build` and `next dev` share the same `.next/` directory: running a build (or `rm -rf .next`) while a dev server is up overwrites/removes chunks the dev server's in-memory webpack runtime still references, producing confusing runtime overlays (`Cannot find module './vendor-chunks/*.js'`, `jest-worker ChildProcessWorker` crashes) that look like code bugs but are cache corruption. `tsc --noEmit` never touches `.next`. Recovery is always: stop the dev server → `rm -rf .next` → restart `npm run dev`. (Node 24 + Next 14.2.x makes the dev worker pool extra fragile here.)

## Architecture — what requires reading multiple files to grasp

### Backend request flow

`server.ts` → `app.ts` (cors + json + `/api` mount) → `routes.ts` (one master router that wires per-module sub-routers under nested paths, e.g. `/projects/:projectId/artifacts`, `/artifacts/:artifactId/relations`) → `modules/<feature>/<feature>.routes.ts` → `<feature>.controller.ts` → optional `<feature>.engine.ts` for pure logic.

Each module is self-contained: routes + controller + (engine when there's non-trivial logic). The same engines are reused by `scripts/seed-demo.ts` to bootstrap the demo, so engines must be pure functions over Prisma — no Express types inside.

ESM gotcha: relative imports inside `backend/src/` use `.js` suffixes (e.g. `from "./auth.controller.js"`) even though the source is `.ts`. This is required by `module: NodeNext` — preserve it when adding new files.

### API envelope (contract — do not break)

Every backend response is either:

```
{ "success": true,  "data": <T>, "message": "..." }
{ "success": false, "error": { "code": "...", "message": "..." } }
```

Wrappers: `utils/response.ts` (`ok`, `created`, `fail`, `HttpError`) on the backend; `lib/api/client.ts` on the frontend unwraps `.data` and throws `ApiError` on failure. Pages call typed wrappers (`projectsApi`, `artifactsApi`, …) from `lib/api/*.ts` — **never `fetch` directly from a page or component**.

### Auth + project access

- JWT bearer tokens. `requireAuth` middleware in `middleware/auth.ts` validates the token and reloads the user from Prisma on every request.
- Project authorization lives in `lib/project-access.ts`, **not** in individual controllers. Use these helpers:
  - `getProjectAccess(projectId, userId)` — returns `{ status, role? }`.
  - `assertProjectRole(projectId, userId, res, minRole)` / `assertCanMutate(...)` — sends the right error response and returns `null` if denied.
  - Role hierarchy (low→high): `VIEWER < DEVELOPER < ARCHITECT < OWNER`. Mutations require `DEVELOPER`+; validation/export require `ARCHITECT`+; member management and project delete require `OWNER`.
- `Project.ownerId` is preserved as the "creator pointer" and is treated as an implicit OWNER membership if no `ProjectMember` row exists — this is the fallback that keeps legacy projects working. Do not remove it.
- Last-OWNER protection: the members API refuses to demote/remove the only remaining OWNER (`LAST_OWNER` error).
- **Public user shape is `toPublicUser` (`auth.controller.ts`) — the single serializer for *every* auth response** (login / register / register-complete / me / updateMe). Add a user field there once and it propagates everywhere; never hand-build a user object in another response.
- **Default workspace.** `User.defaultProjectId` (nullable) is the project the user lands in after login. The FK uses `onDelete: SetNull`, so deleting the project self-heals the preference (no dangling ref). `PATCH /auth/me` accepts it (`null` clears it → dashboard); a non-null value is validated with `getProjectAccess` before saving (`400 INVALID_DEFAULT_PROJECT`). The **post-login redirect** (`(auth)/login`) reads `user.defaultProjectId` → `/projects/<id>` else `/dashboard`. Public pages (landing, `BrandLogo`) deliberately stay on `/dashboard` — they render without `AuthProvider` and can't read the preference (same public/auth split as the accent FOUC). The residual edge (removed from a still-existing default project) falls through to the overview page's "Project unavailable" state.

### Multi-step verified registration + email + rate limiting

The single-step `POST /auth/register` (kept, **deprecated**) is superseded by a verified flow in `modules/auth/registration/`: `POST /auth/register/{start,verify,complete,resend}`. Layered like the rest of the app — pure `registration.engine.ts` (email/name normalization, 6-digit code gen via an *injected* RNG, expiry/cooldown/attempt math, password policy ≥8 + letter + number; **no clock/IO/randomness**) → `registration.service.ts` (Prisma + bcrypt/sha256 + email + JWT; an injectable-deps seam `__setRegistrationDeps` lets tests run against an in-memory fake DB) → thin controller + zod. **Pending state lives in `EmailVerification` (one row per normalized email); a `User` row exists only for a COMPLETED account.** Invariants:

- **No `User` and no JWT until verify + complete.** `start` emails a 6-digit code (bcrypt-hashed, 10-min TTL); `verify` checks it and mints a short-lived **registration token** (sha256-hashed, ~15-min); `complete` consumes the token, creates the `User` (serialized via `toPublicUser`), and issues the JWT. Plaintext codes/tokens are **never persisted**.
- **The attempt cap is atomic.** `verify` reserves an attempt with a conditional `updateMany({ where:{ id, attempts:{ lt: MAX } } })` **before** the bcrypt compare (no TOCTOU under concurrent requests); 5 failures burn the code. The no-record and duplicate paths run a dummy bcrypt so response timing doesn't leak account existence.
- **`start` reveals a completed account on purpose.** It returns **`409 EMAIL_TAKEN`** when a `User` already owns the email (so the wizard stops on step 1 instead of sending a code that never arrives) — a deliberate signup-enumeration tradeoff. **Pending/incomplete registrations still proceed** (no `User` yet); `resend`/`verify`/`complete`/`login` stay neutral/generic. (See `KNOWN_LIMITATIONS.md → Auth`.)
- **Email is one seam.** `modules/email/email.service.ts` (`EmailService`); `getEmailService()` picks by `EMAIL_PROVIDER`: **`resend`** (real, via the Resend REST API with `RESEND_API_KEY` + `MAIL_FROM`), **`smtp`** (placeholder → `503 EMAIL_NOT_CONFIGURED`), or **`dev`** (default; logs a **masked** code, dev only — never logs the code in production). Templates are dark/branded, inline-styled, **HTML-escape the user's name**, and ship both text + HTML. Don't call a provider directly — go through `getEmailService()`. Email config is optional and **not** in `validateConfig()` (the app must boot with zero email creds). The seam also exposes a generic **`sendMail({to,subject,text,html})`** (Dev logs, Resend sends via the shared transport, SMTP throws `503`) for features that render their OWN content (e.g. validation alerts) — build the HTML/text in the feature module and use `sendMail` only for transport; never reach into a provider. `getAppBaseUrl()` (`config/env.ts`; optional `APP_BASE_URL`/`FRONTEND_URL`, never throws) builds absolute links in outbound emails, falling back to a relative route hint when unset. The interface now carries **one method per message type** — `sendVerificationCode`, `sendPasswordResetCode`, `sendEmailChangeCode` (to the new address) + `sendEmailChangeNotice` (to the old address), plus a generic `sendMail` — each with its own template; `ResendEmailService` funnels them all through a **single private `post()`** (one place for the API key + transport-error → `502 EMAIL_PROVIDER_ERROR` / missing-key → `503`). Add a new message type as a method here, never an ad-hoc `fetch`.
- **Rate limiting is in-memory + single-instance.** `middleware/rate-limit.ts` (+ pure `rate-limit.engine.ts`) puts fixed-window limiters on login and each registration route. `clientIp()` uses `req.ip` only — **never raw `X-Forwarded-For`**; set `TRUST_PROXY` when behind a proxy/LB. State resets on restart and doesn't span replicas → move to Redis before any multi-instance deploy.

Frontend: the 4-step wizard is `app/(auth)/register/page.tsx`. It shows server errors **inline in-card** (the login page uses the bottom toast; the divergence is intentional — a toast covers the taller wizard card). All four endpoints have typed wrappers in `lib/api/auth.ts`.

### Forgot-password flow

Verified password reset in `modules/auth/password-reset/`, mirroring registration: `POST /auth/password/{forgot,verify,reset,resend}` (migration `20260605120000_add_password_reset`). **Reuses the pure `registration.engine.ts` helpers** (code gen, expiry/cooldown/attempt math, password policy) — don't duplicate that logic. Layered `password-reset.service.ts` (Prisma + bcrypt/sha256 + email; injectable-deps seam `__setPasswordResetDeps` for in-memory-DB tests) → thin controller + zod. **Pending state lives in `PasswordReset` (one row per normalized email, keyed by email — *not* a User FK — so the request step needs no join), deleted on success.** Invariants:

- **Enumeration-neutral — unlike registration's `start`.** `forgot` returns the **same** `{resendAvailableAt}` whether or not an account exists; an unknown email writes no row and sends no mail but still spends a dummy bcrypt to equalize timing. Email send is **best-effort** (a provider failure is logged + swallowed, never surfaced) so a misconfigured mailer can't make a real account observable. `resend` is likewise neutral.
- **verify → reset, single-use token, no auto-login.** `verify` checks the bcrypt code (atomic attempt reservation *before* the compare, same as registration) and mints a sha256 **reset token** (~15-min); `reset` consumes it, sets the new `passwordHash`, backfills `emailVerifiedAt`, deletes the row, and returns **generic success — deliberately no JWT** (the UI redirects to `/login`). Plaintext codes/tokens never persisted.
- Frontend: 4-step wizard `app/(auth)/forgot-password/page.tsx` (email → code → new password → done), reached from the "Forgot password?" link on login. Typed wrappers `authApi.password{Forgot,Verify,Reset,Resend}`. 22 service tests.

### Verified email-change flow

Authenticated email change in `modules/auth/email-change/`: `POST /auth/email/{request,verify,resend}`, **all behind `requireAuth`** (the actor's `userId` comes from `req.user`, never the body) (migration `20260605140000_add_email_change`). Same engine-reuse + injectable-deps seam (`__setEmailChangeDeps`). **Pending state lives in `EmailChange` (one row per user, `userId @unique`, FK'd to `User` `onDelete: Cascade`).** This is **2-step** (request → verify) — no handoff token, because the target `newEmail` is already known. Invariants:

- **Step-up re-auth + reveals conflicts on purpose (no enumeration concern — the actor is logged in).** `request` re-verifies `currentPassword` (`401 INVALID_CREDENTIALS`), rejects the user's own address (`400 SAME_EMAIL`) and one used by another account (`409 EMAIL_TAKEN`). Unlike forgot-password, send errors are **surfaced** (the user is waiting on their own code).
- **Code to the NEW address; swap only on verify; security notice to the OLD address.** `request` emails the code to `newEmail`; `verify` (atomic attempt cap) swaps `User.email`, sets `emailVerifiedAt`, deletes the row, and **best-effort notifies the OLD address** (`sendEmailChangeNotice`, masked new address — a notice failure must not fail the swap). Uniqueness is re-checked at verify with a `P2002 → 409 EMAIL_TAKEN` race fallback.
- **No JWT reissue.** `requireAuth` authorizes by `userId` and reloads the user every request (the token's `email` claim is never used for authz), so changing the email leaves existing sessions valid; `verify` returns the updated `toPublicUser` and the client calls `setUser`. Don't add a reissue to "fix" the now-stale claim.
- Frontend: the Settings email field is **read-only** — removed from the profile `updateMe` save/dirty path; the **only** UI path to change it is the 2-step `ChangeEmailModal` in `app/(app)/settings/page.tsx` (new email + current password → code). The backend `updateMe` still technically accepts `email`, but nothing in the UI sends it. Typed wrappers `authApi.emailChange{Request,Verify,Resend}`. 19 service tests.

**Shared OTP input.** The 6-digit code box is one component — `components/ui/code-input.tsx` (`CodeInput`) — used by the registration + forgot-password wizards **and** the email-change modal. Fill-only (no auto-submit; the parent owns submit via the `resetSignal`/`invalid` props). Don't re-inline a per-screen copy.

### Prisma client singleton

`lib/prisma.ts` exports one `PrismaClient` instance, cached on `globalThis.__prisma__` so `tsx watch` reloads don't pile up connection pools. Always import from there — do not `new PrismaClient()` anywhere else.

### Backend hardening invariants (security / reliability / ops)

Three guards added in a hardening pass. Each is the *only* sanctioned path for its concern — don't reintroduce the unsafe shortcut they replaced.

- **JWT secret has NO fallback.** `config/env.ts` is the single validated secret source: `getJwtSecret()` throws a `ConfigError` if `JWT_SECRET` is missing, a known placeholder (`change-me-in-production`, `dev-secret-change-me`, the `.env.example` value, …), or shorter than 16 chars; it never logs the value. **Both** `signToken` and `requireAuth` (`middleware/auth.ts`) read the secret through `getJwtSecret()` — they can't diverge. `server.ts` calls `validateConfig()` before `app.listen` and `process.exit(1)`s on `ConfigError`, so the backend **refuses to start** with an insecure secret. Never re-add a `process.env.JWT_SECRET || "…"` fallback; add new required-config checks to `validateConfig()`.

- **Async handler errors reach the central error handler.** `app.ts` imports `express-async-errors` (side-effect, first import) so a rejected/thrown promise in any `async` route handler is forwarded to `next(err)` → `middleware/error.ts:errorHandler` instead of hanging the request. This is why controllers can stay thin `async` functions with no try/catch and still emit the `{success:false}` envelope. `HttpError` mapping is unchanged. Don't remove that import or wrap handlers in redundant try/catch to compensate; the harness in `middleware/error.test.ts` pins the behavior (HttpError → mapped status, generic throw → 500).

- **The error handler never leaks internal messages.** `errorHandler` (`middleware/error.ts`) has two branches: an `HttpError` returns its `status`/`code`/`message` verbatim — **plus its optional `details` when present** (deliberately user-facing context that *only we* populate, e.g. `WEAK_PASSWORD`'s unmet rules, `RESEND_COOLDOWN`/`RATE_LIMITED` retry-after seconds); codes are user-facing (`UNAUTHORIZED`, `NOT_FOUND`, `VALIDATION_ERROR`, `RELATION_EXISTS`, the AI taxonomy, …). **Anything else returns a fixed `500 INTERNAL_ERROR` with the generic message `"Internal server error"` — never `err.message`, a stack, `details`, or DB/Prisma text.** Full diagnostics (timestamp, error name, message, method, request path, stack) go to `console.error` server-side only. `HttpError.details` is safe to forward precisely because it is *only ever set explicitly by us*, never populated from an unexpected error — don't change that, and never pipe `err.message`/`details` from an *unexpected* error into the response to "help debugging" (that reopens an information-disclosure hole); throw a deliberate `HttpError` if a message is meant to be user-facing. `middleware/error.test.ts` pins this (a bare `HttpError` body has no `details` key; generic throw / rejected promise ⇒ `"Internal server error"`, and the raw internal text never appears in the body).

- **Destructive scripts are guarded.** `lib/destructive-guard.ts` (`checkDestructiveSafety` pure fn + `assertDestructiveAllowed` env wrapper) blocks a DB wipe when `NODE_ENV=production` or the `DATABASE_URL` host looks remote/managed (railway, neon, supabase, amazonaws, render, …) — both **hard blocks**; only `localhost`/`127.0.0.1`/`::1` run unguarded. `ALLOW_DESTRUCTIVE_SEED=true` is an escape hatch for an *unrecognized-but-local* host only (it does **not** bypass production/remote). `seed-demo.ts` calls `assertDestructiveAllowed()` as the first line of `main()`; `prisma:reset` is prefixed with `tsx scripts/guard-destructive.ts &&`. Any new wipe/reset script must call the guard before touching the DB. Error prefix: `Refusing to run destructive seed against non-development database.`

### Knowledge graph contract

`ArtifactRelation` is the **single source of truth** for the graph. `/api/projects/:id/graph` only emits artifact nodes — API specs, database models, and diagrams are intentionally **not** native graph nodes. Cross-resource navigation happens via the artifact detail page's "Linked resources" card. Keep this contract intact; the frontend graph code assumes it.

**A directed edge is unique by `(sourceArtifactId, targetArtifactId, relationType)` — enforced at the database level** via `@@unique` (migration `20260601120000_unique_artifact_relation_edge`), not just in application code. Three places must stay in agreement: the DB unique index, the relation controller's pre-check, and the AI bootstrap validator's dedup. The controller wraps the `create` in try/catch and maps a Prisma `P2002` to **`409 RELATION_EXISTS`** (race-safe — the pre-check can lose a race, the DB cannot). Two artifacts may therefore have multiple edges between them only if the `relationType` differs; a duplicate `(source, target, type)` is rejected, never silently created. Don't add a relation-create path that bypasses this `P2002` → `409` mapping.

**Edge uniqueness is DB-enforced.** A directed edge is unique by `@@unique([sourceArtifactId, targetArtifactId, relationType])` — the database, not just app logic, guarantees no duplicate edges (graph SSOT integrity, concurrent-write safety). Uniqueness is **directional and type-scoped**: `A→B` and `B→A`, and two different relation types between the same pair, are all distinct allowed edges. The relation controller maps the `P2002` unique violation to **409 `RELATION_EXISTS`** (race-safe — no pre-check TOCTOU); the AI bootstrap path dedups in its validator *and* catches `P2002` → `BootstrapConflictError`. The Prisma index name is truncated to Postgres's 63-char limit (`ArtifactRelation_sourceArtifactId_targetArtifactId_relation_key`), so hand-authored migrations must match that exact name or `migrate diff` reports drift.

### Validation engine

`modules/validation/validation.engine.ts` is rule-based and **deterministic** — no AI / heuristic-soup logic. It wipes the project's existing `ValidationIssue` rows and recomputes them in one pass, then writes a single `VALIDATED` `VersionEvent`. Rules cover: artifact-relation hygiene, missing docs, security policies, API completeness, DB model integrity, diagram links, churn (≥5 CUDs in 7d), deprecated-but-still-used, single-member projects. Add new rules as pure functions inside the same engine. The four **API Payload** rules (`API_FIELD_UNMAPPED`, `PUBLIC_ENDPOINT_EXPOSES_SENSITIVE_FIELD`, `USER_SCOPED_ENDPOINT_WITHOUT_AUTH`, `RESPONSE_EXPOSES_TOKEN_OR_SECRET`) come from `modules/api-intel/api-validation.ts` (the single payload-heuristics source — see **API Payload Intelligence**) and are mapped to `ValidationIssue` rows here, with the code as a `CODE · ` message prefix (no schema change). The legacy "endpoint marked public on a security spec" rule and the new SECURITY rules share the **`AUTH_ACTIONS` allow-list** (`api-intel/text.ts`) so public auth endpoints (login/register/refresh/verify/forgot-/reset-password) are not flagged. The `MISSING_DOCUMENTATION` rule is the pure `modules/findings/documentation-rule.ts` (Option B — see **Findings, Quick Fix & Relation Remediation**), not inline. Findings are presented with actionable `meta` (rule/why/fix/target/**actions**) by `validation.presenter.ts`, and findings can expose **remediations** — that whole layer is the next section.

**`runValidationForProject` returns `{ issues, newErrorIssues }`** (not a bare `ValidationIssue[]` — a contract its three callers, the controller + both seed scripts, destructure as `{ issues }`). `issues` is the full recomputed set; `newErrorIssues` is the **Option-A dedup** set — OPEN/ERROR findings whose content fingerprint was NOT present before this run — computed by `selectNewErrorIssues` (`validation.status.ts`, reusing the same `issueFingerprint`). The engine stays a pure rule runner with **no side effects**: it only *returns* `newErrorIssues`; the controller dispatches the alert email afterward (see **Validation alert notifications**). Don't move notification/IO into the engine, and keep the engine independent of `modules/notifications/`.

### Findings, Quick Fix & Relation Remediation (`modules/findings/`)

`modules/findings/` is the shared, **deterministic, AI-free** base for finding identity *and* remediation — one place so heuristics can't drift. Each pure file ships a colocated `*.test.ts`; nothing here uses IO/clock/randomness/AI. Files: `finding-types.ts` (vocab), `finding-catalog.ts` (canonical `code → {category, severity, why, suggestedFix, targetKind}` — the single source of a finding's identity), `finding-classifier.ts` (recovers a code from a stored message, since `ValidationIssue` has **no `code` column**), `finding-rules.ts` (the pure architecture findings shared by Validation + Analysis), `documentation-rule.ts`, `finding-actions.ts`, `quick-fix.ts`, `relation-remediation.ts`.

- **MISSING_DOCUMENTATION (Option B)** — `documentation-rule.ts:analyzeMissingDocumentation`. Flags a **documentable** (`SERVICE|API_SPEC|DATABASE_MODEL|SECURITY_POLICY|DOCUMENTATION`), non-`DEPRECATED` artifact that has neither its own `documentationContent` nor an **incoming `DOCUMENTS` relation** (the anti-noise guard: an artifact documented by a linked doc isn't re-flagged — the empty doc is flagged instead). Excludes `EXTERNAL_SYSTEM` + sub-resource types; **DRAFT is in scope** (the UI/AI-bootstrap default — excluding it would hide the findings this rule exists to surface). Keeps the **exact legacy message** for DOCUMENTATION artifacts so prior IGNORED waivers survive; other types get a generic message that still contains `"no documentation content"` so the classifier maps it to `MISSING_DOCUMENTATION`.

- **Action model** (`finding-actions.ts`): `FindingAction = { id, label, kind, status: AVAILABLE|PLANNED|DISABLED, requiresReview?, fixId? }`. `getFindingActions(code, target)` returns a NAVIGATE action (derived from the resolved target) + per-code slots from `FIX_ACTIONS`. A slot is emitted `AVAILABLE` **only when its backing registry agrees** (cross-checked to prevent drift): `getQuickFixIdForCode` for SAFE one-click fixes (`requiresReview` false) or `getRelationRemediationIdForCode` for REVIEW-required ones (`requiresReview` true); otherwise it degrades to `PLANNED` ("Not implemented yet"). `validation.presenter.ts:explainIssue` attaches `actions` to each issue's `meta`; the list endpoint surfaces it.

- **SAFE quick fixes** (`quick-fix.ts` + `validation/quick-fix.controller.ts`): deterministic content, one-click after preview — `GENERATE_DOCUMENTATION_TEMPLATE` (fills empty docs with a fixed section template) and `GENERATE_STARTER_DIAGRAM` (`DIAGRAM_EMPTY` → a `graph TD` starter; only for FLOWCHART/ARCHITECTURE so it can't trip `DIAGRAM_INVALID`). `GET …/quick-fix/preview` (VIEWER+) returns the content; `POST …/quick-fix/apply` (ARCHITECT+) writes it with an **atomic no-clobber `updateMany`** (only while still empty → `409` otherwise — matching the registration attempt-cap pattern), records `VersionEvent(origin:"QUICK_FIX")`, **re-runs validation**, and returns the refreshed enriched issues. Content is re-derived server-side from the `fixId`; the client preview is never trusted.

- **REVIEW-required relation remediation** (`relation-remediation.ts` + `validation/relation-remediation.controller.ts`) — never auto-applied. `DIAGRAM_UNLINKED → LINK_DIAGRAM_ARTIFACT` (sets `diagram.artifactId`, an FK, **not** an edge), `SECURITY_POLICY_NOT_LINKED → LINK_SECURITY_POLICY` (creates a `SECURES` relation), `ORPHAN_ARTIFACT → LINK_ORPHAN_ARTIFACT` (creates a relation, type per candidate). `DEPENDS_ON_DEPRECATED` / `HIGH_FAN_OUT` / `HIGH_CHURN` are **navigation-only** (PLANNED — architectural/process decisions, never auto-fixed). `GET …/remediation/preview` returns deterministic scored candidates; `POST …/remediation/apply` (ARCHITECT+) **re-derives candidates server-side and applies ONLY a selection the generator actually produced** (no client-injected target), via atomic diagram-link or a **mirrored `ArtifactRelation` create** (`P2002 → 409 RELATION_EXISTS`, self/cross-project guards — parallel code to `relations.controller.ts`, keep in sync), records `VersionEvent(origin:"QUICK_FIX", source:"RELATION_REMEDIATION", …)`, re-runs validation. This honours AI-Safety Rule 4: validation stays deterministic — a fix repairs the underlying resource and *re-runs* the engine; it never edits `ValidationIssue` rows directly.

- **Confidence & evidence engine** (the V2 scoring core): a candidate is `{ targetId, targetTitle, targetType, relationType?, score (0–100), confidence, evidence[] }`. Additive weights (`EVIDENCE_WEIGHTS`): `TITLE_MATCH 40, MERMAID_NODE_MATCH 30, PHRASE_TITLE_MATCH 25, API_INTELLIGENCE 25, TOKEN_MATCH 20, EXISTING_NEIGHBORHOOD 15, ARTIFACT_TYPE_COMPATIBILITY 15`; `score = min(100, Σ)`; bands `80+ HIGH / 50–79 MEDIUM / 0–49 LOW` (`confidenceFromScore`). Sort is `score desc → confidence → title asc → id`. **Per-finding priorities are load-bearing:** diagram prefers a Mermaid node match (diagram-title similarity is capped at the TOKEN tier; `PHRASE_TITLE_MATCH` — candidate title contained as whole words in the diagram title — is **node-gated** so a title-only candidate can never outrank a drawn node, since `TOKEN 20 + PHRASE 25 = 45 > MERMAID 30` otherwise); security prefers an inferred `SECURED_BY` edge over title; orphan adds neighborhood + type-compat and **respects relation direction** (DOCUMENTATION/SECURITY_POLICY targets are skipped — a `DOCUMENTS` edge goes doc→thing, never orphan→doc). A candidate is emitted **only with ≥1 real signal** (title/token/mermaid/api) — never a random typed/securable artifact. `isManualFallback(candidates)` (= no candidate reaches ≥50) is the **single source of truth** used by every preview branch (don't reinline it per-branch — that once diverged); the UI still lists low candidates (never auto-hidden) plus an Open-artifact/Open-graph manual path. Inferred edges come from `api-intel` (`analyzeProjectApiIntel().inferredEdges`).

- **Frontend** (`app/(app)/projects/[projectId]/validation/page.tsx`): the expanded issue panel's "Available actions" renders NAVIGATE (link), SAFE → **Preview Fix** modal (generated content), REVIEW → **Review Fix** candidate-picker modal (confidence badge + `NN/100` + expandable "Why suggested?" evidence; Apply gated on an explicit selection; low candidates remain selectable). The **sidebar Validation badge** is kept live by a tiny Zustand store `lib/validation-counts.ts` — the validation page publishes its open count on every change, the sidebar seeds it from the server count on navigation; otherwise the badge went stale after a fix. (Public/auth split conventions still apply.)

- **Testbed** (`scripts/seed-payload-testbed.ts`, `npm run seed:testbed`): seeds one fixture per path — `Empty Documentation Test` (MISSING_DOCUMENTATION), `Empty Diagram Test` (DIAGRAM_EMPTY), `Undocumented Service Test` (Option-B MISSING_DOCUMENTATION), `Billing Security Policy` (SECURITY_POLICY_NOT_LINKED), the `Billing Service Architecture` unlinked diagram (DIAGRAM_UNLINKED — note the diagram **record title** drives `TOKEN`/`PHRASE` scoring, not the Mermaid), and `Patient Portal` vs `Notification Service` (ORPHAN with candidates vs manual fallback). The seed prints each path's candidates as a self-check.

### User settings & notification preferences (`modules/settings/`)

Per-user **account** settings (distinct from the per-browser workspace tweaks in the Zustand store). `GET`/`PATCH /settings/notifications` (mounted at `/settings` behind `requireAuth`) read/write the caller's OWN row only — there is **no `userId` in the body**; it comes from `req.user`. Backed by `UserNotificationPreference` (one row per user, FK `onDelete: Cascade`), **lazily created on first PATCH** — an absent row means "all defaults" (both flags `false`), so reads never create a row. The pure `settings.engine.ts` owns the shape: defaults, row→DTO projection, and a **partial merge** (a PATCH carrying one flag never clobbers the other). MVP flags: `emailDigestEnabled` (persisted intent only — **no digest sender exists**) and `validationAlertsEnabled` (wired — see next section). Frontend: the Settings → Notifications tab (`app/(app)/settings/page.tsx`) toggles optimistically via `lib/api/settings.ts` and reverts on error; Slack stays a disabled `SOON` row.

### Validation alert notifications (`modules/notifications/`)

The first real notification path and a strict **side-effect layer** off the deterministic validation core: the engine computes + returns the new ERROR findings, the **controller** dispatches the email, and the notification code never reads or writes validation/analysis state. `sendValidationAlerts` (`validation-alert.service.ts`) **never throws** — any failure (recipient DB read, provider not configured, transport error) is logged (a code only, never the provider's raw response or any secret) and swallowed, so a notification problem can never fail the validation run, roll back issues, or skip the `VersionEvent`. Invariants:

- **Recipients = project OWNERs.** The owner pointer (`Project.ownerId`, implicit OWNER) + any `ProjectMember` with role `OWNER`, deduped (pure `selectAlertRecipientUserIds`). ARCHITECT / DEVELOPER / VIEWER are **not** alerted. (Documented decision: `ProjectRole` has no ADMIN tier — ADMIN is a global *user* role — so the spec's "OWNER/ADMIN" maps to project OWNERs.)
- **Opt-in, one email per run per user.** Only recipients with `validationAlertsEnabled = true` are emailed, once each — **one summary email per run, never one per issue**.
- **Dedup is Option A** (no schema change): only the engine's `newErrorIssues` (ERROR/OPEN, fingerprint absent before this run) alert. A still-open, waived (IGNORED), or already-seen ERROR (any prior status) does **not** re-alert, so reruns over the same unresolved findings don't spam.
- **Content in the module, transport via the seam.** `buildValidationAlertEmail` (pure) renders the dark/branded HTML+text (subject `Minotaurus validation alert: {project}`, count, top-5 messages with the `CODE · ` prefix stripped, Validation-page link via `getAppBaseUrl()`, the "Validation alerts are enabled in your settings" footer; user input HTML-escaped); delivery goes through `EmailService.sendMail`, never a provider directly. `sendValidationAlerts` takes an injectable deps seam (recipient loader, email service, logger, base URL) so it's unit-tested without a DB or network (`validation-alert.service.test.ts`).
- **Out of scope (deliberately not built):** weekly digest sending, Slack, cron, queues, notification history. Email always goes to the user's verified account address.

### Artifact title uniqueness

Titles are unique **per project**, case-insensitive and whitespace-normalized (trim + collapse internal whitespace + lowercase). Enforced two ways and both must stay in sync:

1. DB: `Artifact.normalizedTitle` column + unique index `(projectId, normalizedTitle)`.
2. Controller: pre-check in create/update (and in the ingestion `CREATE_NEW` confirm path), returning 409 `ARTIFACT_TITLE_TAKEN`.

This applies **only** to `Artifact`. API specs, diagrams, database models, etc. allow duplicate titles within a project.

### Version events

Every CUD across the platform (artifacts, relations, API specs, endpoints, DB models/entities/fields, diagrams, documentation, members, validation runs, ingestion confirms) appends a `VersionEvent` via `modules/versions/versions.engine.ts:recordVersionEvent`. This feeds the version history timeline, the dashboard "Recent changes" widget, and the impact-analysis recent-events list. New mutations must record their event.

### Ingestion

`modules/ingestion/` has one controller plus four parser engines (`markdown`, `openapi`, `mermaid`, `sql`). Flow: draft → parse → confirm. Confirm uses the same artifact / API spec / diagram / DB model creation paths as the regular controllers (including the artifact title check) so ingestion never bypasses validation. Ingestion is an audit log — deleting an `IngestionRecord` does **not** cascade-delete the resources it produced (no FK on the join). UI copy reflects this ("Remove log" vs "Delete draft").

### AI Bootstrap Wizard (`modules/ai/`)

The first AI feature (see also **AI Architecture Review** below). From an existing **empty** project, the user describes an idea → AI proposes a draft → the user reviews/selects → confirmed items are applied through the **existing** artifact/relation/diagram creation paths. It implements the `## AI Safety & Determinism Rules` above: **AI proposes; a human-gated deterministic step disposes.** Code in `modules/ai/` never calls `prisma.*.create/update/delete`.

**Flow & files.** `ai.routes.ts` mounts `POST /projects/:projectId/ai/bootstrap/{propose,apply}` (both `DEVELOPER`+). `ai.controller.ts` is thin (role check + envelope). `ai.service.ts:proposeBootstrap` orchestrates: build prompt → provider → Zod-parse (one repair retry) → normalize Mermaid → deterministic preview validation → persist an `AiSession`. `proposal/bootstrap.apply.ts:applyBootstrap` is the **only** path to the DB: it re-validates the (user-edited) proposal server-side, then creates artifacts/relations/diagrams inside a `$transaction`, re-deriving the same fields the regular controllers use (title normalization, `DRAFT` status, `description:""`, Mermaid validation) rather than importing those controller functions, records a `VersionEvent` per entity with `metadata.origin:"AI"`, and updates the `AiSession`. (Apply replicates the controllers' invariants but is parallel code, not a call into them — keep the two in sync when a controller's creation contract changes.)

**Stateless proposal + audit.** The proposal is **not** persisted as a source of truth — the client holds it between propose and apply, and apply re-validates from scratch (never trust the client). `AiSession` (Prisma) is **lightweight audit metadata only** — like `IngestionRecord`, never a graph node — recording idea, model, tokens, status (PROPOSED/APPLIED/DISCARDED), counts, and the proposal snapshot.

**The generation contract is expressed twice, derived from the Prisma enums** (`proposal/bootstrap.schema.ts`): a Zod schema (validates Claude's output at propose and the user subset at apply) and a Claude tool `input_schema` (forces conforming JSON via `tool_choice`). **Field/emit order is load-bearing:** `summary → artifacts → diagrams → relations → databaseModels → apiSpecs` — the required `diagrams` come before `relations`, and the newest **optional** blocks (`databaseModels`, then `apiSpecs`) emit last, so a truncated response sheds API specs, then database models, then relations, before ever losing the required diagrams.

**Artifacts are created `DRAFT`; AI prose never lands on entity fields** — rationale/confidence live only in the proposal/audit snapshot, not in `Artifact.description` (kept `""`). Confidence is advisory (review UI pre-checks ≥0.6); it never gates apply.

**AI Mermaid is structure-only.** `prompts/bootstrap.prompt.ts` forbids styling; `proposal/mermaid-normalize.ts` strips any `classDef`/`class`/`style`/`linkStyle`/`:::class`/`%%{init}%%` at propose + apply (the validator parses the normalized form, so styling never causes rejection). The shared renderer owns appearance: `mermaid-preview.tsx:forceLabelVisibility` darkens near-white shape fills (luminance-based) as a **runtime fallback** for user-pasted / imported styled Mermaid — not the primary strategy. Don't reintroduce AI-side styling.

**Diagram ↔ artifact referential integrity.** The validator checks that every node a proposed diagram draws resolves to a selected/existing artifact — the **same `resolvable` set relations use** — by reusing the parser's `nodeHints` (no second extractor). A diagram referencing an unselected/missing artifact is rejected (skipped) at apply and warned live at review; it is never silently applied (SSOT stays authoritative). **External systems are modeled as `EXTERNAL_SYSTEM` artifacts** (Model A) — there are no Mermaid-only nodes — and the prompt requires explicit bracketed node labels matching artifact titles exactly. (This also fixed a latent `NODE_LABEL_RE` bug in `ingestion/mermaid.engine.ts` where an edge arrow `-->` was swallowed by the asymmetric-flag node shape.)

**Provider & token budget (`providers/`).** `ai.provider.ts` is the only LLM seam; `ai.provider.anthropic.ts` calls **non-streaming** `messages.create` with forced tool use and a cached system block (`AI_MODEL` default `claude-sonnet-4-6`, `AI_MAX_TOKENS` default `8192`). `StructuredResult` preserves `stopReason`/`usage`/`model`/`maxTokens`/`durationMs` — **do not discard them**; that lossiness once made a truncated response look like a provider failure.

**Error taxonomy (honest, not generic).** A large proposal can exhaust `max_tokens` → Anthropic returns **200 with `stop_reason:"max_tokens"`** and a complete-but-partial tool object. The service treats this as truncation, **not** a provider failure: it throws `AiOutputTruncatedError` → **422 `AI_OUTPUT_TRUNCATED`** (`details:{maxTokens,outputTokens,suggestion}`) and **skips the repair retry** (it would truncate identically). Codes: `503 AI_NOT_CONFIGURED`, `502 AI_PROVIDER_ERROR` (transport / no tool block), `422 AI_OUTPUT_TRUNCATED`, `502 AI_SCHEMA_ERROR` (complete but off-schema after one retry); apply adds `422 AI_VALIDATION_FAILED` / `409 AI_APPLY_CONFLICT`. Every failure logs a scalar `[ai] bootstrap proposal failed {…}` line (projectId/userId/model/stopReason/tokens/durationMs/code) — **never** the prompt, AI output, or secrets.

**Determinism untouched, scope bounded.** AI lives outside the deterministic core — the validator and apply are deterministic, and the analysis/validation engines are neither consulted nor modified.

**Bootstrap V2 — extended generation (database + API).** The wizard now also proposes **database models** (Phase 1) and a lightweight **API catalog** (Phase 2) through the *same* propose → validate → review → confirm path. The original `artifacts + relations + 1–3 diagrams` core is unchanged; both new blocks are **optional**, emitted last, and `.default([])` in the Zod schema, so a DB-less / API-less / truncated response still parses. Both follow every rule above (proposed-only until confirm, `description:""` on entities, validator↔apply parity, a `VersionEvent` with `origin:"AI"` per created row). Whole-block selection only in the review UI (no per-entity/per-endpoint checkboxes yet).

- **Database (`databaseModels[] { entities[] { fields[] } }`).** Structure-only: names, `databaseType`, PK/FK flags — never prose. The FK target is **`referencesEntityName`** (an entity name in the *same* model); the validator resolves it **deterministically in two passes per model** — pass 1 freezes the structurally-valid entity set (named, unique-in-model, ≥1 field), pass 2 resolves each field's FK against that frozen set (self-reference allowed; unknown target → that field rejected, the model still applies). Apply mirrors it: create all entities first → build a `name→id` map → create fields mapping `referencesEntityName → referencesEntityId` (`isForeignKey = given || !!ref`). Optional `artifactTitle` links a model to its owning artifact via the same `resolvable` set relations/diagrams use (dropped, not rejected, if unresolved). Caps: ≤4 models / ≤8 entities / ≤12 fields.

- **API catalog (`apiSpecs[] { endpoints[] }`).** Catalog only — method + path + one-line summary; **`requestSchema`/`responseSchema` are NOT generated** (persisted `""`) to bound token cost / truncation. `method` (strict `HttpMethod` enum) and endpoint `summary` (≤120) are strict in Zod — a malformed endpoint trips the one repair retry; the validator then skips endpoints with a bad method / non-`/` path / empty-or-overlong summary / duplicate `(method,path)`, rejects a spec with no surviving endpoint (`"API spec has no valid endpoints"`), and skips duplicate spec titles. `artifactTitle` links best-effort like DB models. Caps: ≤4 specs / ≤10 endpoints.

- **Read-only payload inspection (frontend counterpart).** The endpoint table on the API spec detail page (`app/(app)/projects/[projectId]/api/[apiSpecId]/page.tsx`) renders each endpoint's `requestSchema`/`responseSchema` in an **expandable detail row** — payloads are inspectable without opening the Edit modal (Edit stays edit-only; its CRUD is unchanged). A pure `formatSchema` helper pretty-prints valid JSON (objects/arrays, 2-space) and falls back to verbatim monospace for free text; empty schemas show "No request/response schema defined" — which is exactly what an AI-bootstrapped spec (schemas persisted `""`, per the bullet above) shows until a user fills them in. Expand/collapse is a CSS `grid-template-rows: 0fr → 1fr` transition (no height measurement) plus a rotating chevron, both gated by `motion-reduce:transition-none`; the row-click toggle and a real chevron `<button>` (with `aria-expanded`) both drive it, and Edit/Delete `stopPropagation` so they don't expand the row.

- **Parity + audit.** Apply is parallel code, not a controller call (same as artifacts/diagrams), so the database-models / api-specs controller creation contracts (`isForeignKey` derivation, same-model FK scoping, `version` default `1.0.0`, empty schema bodies) must stay in sync. `AiSession` carries `*Proposed`/`*Created` counts per type (migrations `…_ai_bootstrap_db_counts`, `…_ai_bootstrap_api_counts`). The export SSOT snapshot → analysis → PDF chain and AI Review already read DB/API models from the snapshot, so they needed no change. **Phase-3 (documentation generation, security policies) remains future work**, added the same way.

### AI Architecture Review — Full Review + Advisor (`modules/ai/review/`, `modules/ai/advisor/`, `modules/ai/architecture/`)

The first **read-only** AI feature family: AI *interprets* the deterministic analysis, never produces or mutates it. It is **one user surface ("AI Review") with two modes**:
- **Full Review** (`modules/ai/review/`) — the comprehensive architecture audit. Answers *"How healthy is this architecture?"*
- **Advisor** (`modules/ai/advisor/`) — a prioritized action plan, **coach not auditor**. Answers *"What should I do next?"*

Both share the same one-directional, AI-free chain up to the model call:

```
SSOT → buildExportContent → analyzeExportSnapshot (AnalysisResult) → buildReviewDigest → AI {review|advisory}
```

`AI → AnalysisResult` is forbidden; `AnalysisResult → AI` is the whole feature (Safety Rule 3).

**Consolidation & the shared runner (`modules/ai/architecture/`).** The two modes are de-duplicated onto one engine (they used to be near-identical copies — see the overlap audit). `analysis-runner.ts` owns the deterministic context (`buildAnalysisContext` / `computeAnalysisAndHash`, the 9 `ARCH_ANALYSIS_SECTIONS`, `hashAnalysis`) **and** the provider generation loop (`runArchitectureGeneration<T>` — forced tool, one repair retry, truncation salvage, full error taxonomy). `generate.ts` is the single typed entry point `generateAiArchitectureAnalysis({ mode: "REVIEW" | "ADVISOR", projectId, userId })`; the mode selects the per-mode `GenerationSpec` (prompt / schema / verification policy / output sections) + persistence kind. Each mode service holds only its spec + persistence + read endpoints. The digest (`review.digest.ts`) and evidence allow-list are shared, so both modes cite the same deterministic facts and the heuristics can't drift.

**Full Review — flow & files.** `ai.routes.ts` mounts `POST /projects/:id/ai/review` (generate; `DEVELOPER`+) plus three **read-only** GETs that reuse persisted reviews with **no AI call**: `/review/latest`, `/reviews` (history), `/reviews/:reviewId`. `review.service.ts:generateArchitectureReview` builds the analysis context, runs the shared `runArchitectureGeneration` with the review `GenerationSpec`, applies `verifyReviewEvidence`, and persists an `AiSession(REVIEW)`. The GETs recompute the analysis (deterministic, cheap) only to compute staleness + the score cards.

**The digest is built from `AnalysisResult`, never the raw SSOT** (`review.digest.ts`, pure/deterministic). It reads every metric from the analysis and only a per-type/status **count histogram** from the snapshot (the analysis exposes coverage %, not raw counts). All lists are capped (default 10) with their true totals kept. It also emits `evidenceKeys` — a deterministic allow-list of citations (metric paths + the ids of *shown* items only).

**Bounded output (truncation root-cause fix).** `review.schema.ts` caps item counts (strengths 3, risks 5, blindSpots 3, governanceReview 3, validationCommentary 3, recommendations 5) so total output stays ~3–5k tokens at any project size. **Field/emit order is load-bearing:** `executiveSummary → … → recommendations` (last, most expendable on truncation). The prompt + tool schema state the prose targets (≤280 / summary ≤700); the **Zod gate has a small buffer** (prose ≤400, summary ≤900) so a marginal overshoot is accepted instead of failing the whole review. `AI_REVIEW_MAX_TOKENS` (default 12000) is headroom, not the primary guard.

**Explainability is verified, not trusted.** `review.verify.ts:verifyReviewEvidence` strips any `evidence.ref` not in `digest.evidenceKeys`; a finding left with zero evidence is flagged `unverified` so the UI shows it as advisory-but-unsupported. AI explains; a deterministic check disposes.

**Graceful truncation.** On `stop_reason:"max_tokens"`, `review.salvage.ts` salvages the completed prefix (recommendations emit last) and returns `truncated:true` + `missingSections[]`; only if nothing usable arrived does it surface `422 AI_OUTPUT_TRUNCATED`. Never present incomplete output as complete. Other codes mirror bootstrap (`503 AI_NOT_CONFIGURED`, `502 AI_PROVIDER_ERROR`, `502 AI_SCHEMA_ERROR`).

**Persistence & staleness.** `AiSessionKind` gained `REVIEW` and `AiSession` gained a nullable `analysisHash` column (migration `20260531234232_ai_review_session`). A review is audit metadata only (like a bootstrap session): `kind:REVIEW`, `analysisHash`, and `proposal` JSON `{ review, generatedAt, truncated, missingSections, verification }`. `review.read.ts:hashAnalysis` **excludes `meta.generatedAt`** so the hash fingerprints project *state*, not assembly time (otherwise every recompute looks stale); `toStoredReviewResult` maps a stored row → `ReviewResult` and computes `stale` vs the current hash. History is preserved — each generate is a new row; nothing is overwritten.

**Frontend (one page, two modes).** `app/(app)/projects/[projectId]/review/page.tsx` is the single "AI Review" sidebar item; there is **no separate Advisor nav item or page** (the legacy `/projects/:id/advisor` route is a redirect to `…/review?mode=advisor`; the topbar breadcrumb is "AI Review"). A segmented control switches **[ Full Review | Advisor ]**, synced to the `?mode=advisor` query param so a refresh keeps the tab. Each mode lazily loads its latest persisted result on first activation (**no AI call**), shows an empty state when none, a status block (generated / model / `Current` vs `Project changed since this {review|advisory}`), a mode-aware read-only history dropdown, and a manual generate button. Regeneration is always explicit — never automatic (cost / latency / auditability). **Full Review** shows the deterministic score-card grid (`number + GRADE_BANDS band label`, bottom-border progress strip via `scoreColorVar`, existing `--c-*` palette) + the six audit sections. **Advisor** renders as a roadmap (see below), no score grid.

**Advisor mode (`modules/ai/advisor/`) — coach, not auditor.** Deliberately differentiated from Full Review so users never read the same thing twice. The Advisor output is **only**: `executiveSummary` (a ≤80-word *snapshot*), `focusAreas` (top 2–3 concerns, max 3), `opportunities` (max 4), and `recommendations` (max 5, `{title, priority, rationale, evidence[]}`). It does **not** emit Full Review's audit sections (strengths / risks / blind spots / governance / validation commentary) — `advisor.prompt.ts` reframes hard ("you are NOT performing an audit; prefer prioritization over coverage, actionability over completeness"). Emit order is load-bearing (`executiveSummary → focusAreas → opportunities → recommendations`, recommendations last/most-expendable on truncation), and the budget override is `AI_ADVISOR_MAX_TOKENS` (default 6000).

- **Stricter verification policy:** `advisor.verify.ts` *discards* any item with no surviving evidence (Full Review *flags* it `unverified`); recommendations are evidence-mandatory, deterministically sorted HIGH→MEDIUM→LOW, capped at 5.
- **Persistence & reads mirror Full Review:** `AiSessionKind` gained `ADVISOR` (migration `20260607120000_ai_advisor_kind`). `advisor.service.ts:generateArchitectureAdvisory` persists an `AiSession(ADVISOR)` (`proposal` JSON `{ report, generatedAt, truncated, missingSections, verification }`); `POST /ai/advisor` generates, `GET /ai/advisor/latest` / `/advisors` / `/advisors/:advisorId` reuse persisted advisories with **no AI call**. `advisor.read.ts:toStoredAdvisorResult` normalizes the stored report **field-by-field to the current shape** (a row written under an earlier contract reads back with empty arrays, never `undefined`) and computes `stale` vs the current hash. The service exposes a `__setAdvisorDeps` injection seam (db / provider / context / hash) so persist/read/staleness are unit-tested with in-memory fakes (no DB, no network).
- **Read-only w.r.t. architecture:** the Advisor writes **only** its own `AiSession(ADVISOR)` record (allowed — it stores AI output, not architecture state) and has no apply path. `advisor.no-mutation.test.ts` asserts at the source level that no advisor file writes any model other than `aiSession`.
- **Frontend roadmap UI:** no score-card grid (a compact health line keeps the snapshot lightweight); **Current Focus Areas** and **Recommended Next Steps** are the visually dominant `RoadmapSection`s (accent-bubbled header, numbered next-steps), with boxed opportunities. Every category item is a box (`NoteBox`).

### AI Documentation Assistant (`modules/ai/documentation/`)

Per-artifact, on-demand AI drafting of an artifact's Markdown documentation — **not** part of AI Bootstrap (which deliberately does **not** generate docs) and not bulk generation. It's the third read-only AI feature on the Safety rules: **AI drafts text; the user reviews/edits and saves through the existing path.** Code in `modules/ai/documentation/` never calls `prisma.*.update` on `documentationContent` — the only persistence is the unchanged `PUT /artifacts/:id/documentation` the user triggers by clicking **Save**.

**Flow & files.** `ai.routes.ts` mounts `POST /projects/:projectId/ai/documentation/artifacts/:artifactId/draft` (`DEVELOPER`+ — the user must be able to save what they draft). `doc-draft.controller.ts` is thin (role check + envelope + AI error mapping). `doc-draft.service.ts:generateDocumentationDraft` orchestrates: load the bounded neighborhood → `buildArtifactDocumentationDigest` → provider (forced tool, one repair retry) → Zod-parse → best-effort `AiSession(DOCUMENTATION_DRAFT)` audit row. It returns a **draft only** (`{ sessionId, markdown, mode, generatedAt, model, usage, truncated }`); the frontend opens the existing `DocumentationEditor` pre-filled with `markdown` and the user edits + Saves.

**Bounded context, never the whole project (`doc-draft.digest.ts`, pure/deterministic).** The digest holds only: project name/description; the one artifact (title/type/status/tags/description + existing-doc length & 280-char excerpt); its incoming/outgoing relations (with neighbor title/type/status, capped 20 combined); and resources linked via `artifactId` — API specs (title/version/endpoint count/top 5 paths), DB models (title/type/≤12 entity names), diagrams (title/type), and **open** validation issues (severity/category/message, capped 10). Every list keeps its true `total`. No full Mermaid, no full API schemas, no full DB fields. The service does the Prisma joins; the builder is a pure capping/shaping function (unit-tested without a DB).

**Output contract (`doc-draft.schema.ts`).** A single `markdown` string — Zod gate ≤4000 chars (prompt target ~1200–1800; the gate is a buffer, the user edits anyway), plus a Claude tool `input_schema` forcing `{ markdown }`. `doc-draft.prompt.ts` fixes the section skeleton (Purpose / Responsibilities / Dependencies / Related APIs / Related Data / Risks·Notes / Open Questions), forbids invented facts / Mermaid / marketing language, and requires gaps to be phrased as Open Questions.

**Existing docs are never overwritten by AI.** When the artifact already has documentation, the digest flags it and the result `mode` is `"replacement_suggestion"` (vs `"new"`); the editor opens the AI draft as the editable buffer while the saved content stays untouched until the user clicks Save. The frontend button reads **"Generate with AI"** when empty, **"Improve with AI"** when docs exist (empty-state pair is an equal-width `grid-cols-2`).

**Error taxonomy & truncation.** Mirrors bootstrap/review: `503 AI_NOT_CONFIGURED`, `502 AI_PROVIDER_ERROR`, `502 AI_SCHEMA_ERROR`, `404 NOT_FOUND` (artifact not in project). On `stop_reason:"max_tokens"` a non-empty Markdown string is salvaged and returned with `truncated:true` (a draft is advisory); only an unusable truncation surfaces `422 AI_OUTPUT_TRUNCATED`. Budget override: `AI_DOC_DRAFT_MAX_TOKENS` (default 2048). Every failure logs the same scalar `[ai] documentation draft failed {…}` line — never the prompt or AI output.

**Audit.** `AiSessionKind` gained `DOCUMENTATION_DRAFT` (migration `20260602100000_ai_documentation_draft_kind`). The audit row is best-effort metadata only — `idea:""`, model/tokens, and `proposal` JSON `{ artifactId, generatedAt, mode, truncated, markdownLength }`; **never** the drafted Markdown itself or the prompt. A failed audit write must not fail the read-only draft.

### API Payload Intelligence (`modules/api-intel/`)

**`modules/api-intel/` is the SINGLE deterministic source for everything derived from endpoint payloads** — payload/field extraction, entity matching, workflow inference, validation rules, knowledge-graph inferred edges, analysis metrics, PDF output, and AI-review evidence all read from this one module, so the heuristics can never drift. It is **non-AI, pure, and deterministic** (no `Date.now()`, no randomness, no IO; every output sorted ⇒ same input gives a deep-equal result), it **never writes the DB**, and it **never persists** anything it infers (no `ArtifactRelation` is ever created). It is the deterministic counterpart to the AI layer: AI *explains*, api-intel *computes*.

**Files (pure unless noted).** `text.ts` (string heuristics: `idStem`, `parsePath`, `normalizeToken`, `singularize`, `AUTH_ACTIONS` + `isAuthActionPath`); `api-intel.constants.ts` (credential/PII tokens, generic-field stoplists, action-verb lexicon); `api-intel.types.ts` (all DTOs); `payload-analyzer.ts` (the core: `extractFieldNames`, `sensitiveKind`, `buildEntityMatcher`, `matchEntities`, `buildArchitectureLinks`, `buildInferredEdges`, `analyzeProjectApiIntel`); `workflow-infer.ts` (`inferWorkflow`); `api-validation.ts` (`analyzeApiValidation` — the four rules); `api-metrics.ts` (`analyzeApiIntelCounts`); thin `api-intel.controller.ts` + `api-intel.routes.ts` (read-only **`GET /projects/:projectId/api-intel`**, VIEWER+). Each pure file ships a colocated `*.test.ts`.

**Three-tier inference (per endpoint; every output confidence-graded with a mandatory `basis`).**
- **Tier 1 — payload → entity (`medium`).** `extractFieldNames` pulls ALL request+response keys (JSON keys recursively, else free-text tokens); id-like fields (`patientId`→`patient`) and field-name matches (`email`→an entity with an `email` field) resolve to a `DatabaseEntity`. **Tie-breaking is context-aware:** when an ambiguous field name exists on several entities (`email` on both `User` and `Patient`), the matcher prefers entities in DB models **reachable 1-hop from the spec's linked service artifact via real `ArtifactRelation` edges** (so an Authentication API prefers `User`/`Session`); with no such context it falls back to the deterministic alphabetical first. Context **only breaks ties** — never changes an unambiguous match, and the no-context path is byte-identical to the old behavior.
- **Tier 2 — real SSOT relations (`high`).** From the **primary** entity's model artifact (+ name-matched anchors), walk existing `ArtifactRelation` edges: `SECURES`→Security, `DOCUMENTS`→Documentation, `USES`/`DEPENDS_ON`/…→Related. These aren't guesses — they're confirmed relations, contextualized to the endpoint (the same traversal as the artifact `analyzeImpact`). **Cross-referenced entities do NOT anchor the walk** (a `patientId` on an appointment endpoint must not inherit Patient's docs/policies) — they still appear under "Touches", just not as implementing/documenting/securing artifacts.
- **Tier 3 — name fallback (`low`).** Resource/parent + **primary-entity** title-token match against typed artifacts (`SERVICE`→Related, `SECURITY_POLICY`→Security, `DOCUMENTATION`→Documentation). Deliberately **NOT** every matched entity name — that once propagated docs/policies too aggressively.

**`payloadFields` vs `referencedFields`.** `payloadFields` = every extracted field (the complete payload surface, UI "Payload Fields"); `referencedFields` = the subset that drove inference (id-like or entity-matched, UI "Referenced Fields"). A field with no entity column (e.g. `firstName`) appears in `payloadFields` only.

**Workflow inference (`workflow-infer.ts`).** Deterministic rules over `(method × parsed path × payload fields × matched entities)` → `Creates/Updates/Deletes/References/Reads/Authenticates/Generates/Starts/Triggers/Requires/Handles` signals, each with `confidence` + an explainability `basis`. Action keywords (`login`→Authenticates, `register`→Starts Onboarding) layer on the method-derived CRUD; `nonCrud` actions suppress the generic create. A **public** endpoint taking a credential reads **"Handles Credentials"** (it establishes a credential — it does not "require authentication").

**Validation rules (`api-validation.ts` → `validation.engine.ts`).** Four deterministic codes — `API_FIELD_UNMAPPED` (INFO/API), `PUBLIC_ENDPOINT_EXPOSES_SENSITIVE_FIELD`, `USER_SCOPED_ENDPOINT_WITHOUT_AUTH`, `RESPONSE_EXPOSES_TOKEN_OR_SECRET` (SECURITY) — produced by the pure analyzer and mapped to `ValidationIssue` rows by the existing deterministic engine. **No schema change: the code is a `CODE · ` message prefix** (the model has no `code` column). The security rules share the **`AUTH_ACTIONS` allow-list in `text.ts`** (login/register/refresh/verify/forgot-password/reset-password) so public auth endpoints don't false-positive; the **same allow-list now also gates the pre-existing `isSecuritySpec` "marked public" rule**. AI never writes validation state (Safety Rule 4).

**Knowledge-graph inferred edges (`buildInferredEdges`).** Per-spec aggregation → **artifact-level** edges (`TOUCHES` spec-artifact→data-model, `SECURED_BY`, `DOCUMENTED_BY`), **deduped against real `ArtifactRelation` edges** so the overlay only ADDS connections the solid graph can't show. Rendered dashed + "inferred"-badged, **toggle-gated, never persisted, no new graph node types** — the artifact-only graph contract is intact. The graph page exposes edge visibility via a single **"Edges" dropdown** over the four real×inferred states (`Real relations` (default) / `Inferred links` / `Real + inferred` / `No edges`); header counts stay honest totals (X artifacts · Y relations · Z inferred). Hiding real edges affects rendering only — the dagre layout always reads the full real relation set (see *Knowledge graph rendering → Layout*).

**Analysis / PDF / AI-review metrics (`api-metrics.ts` → `AnalysisResult.apiIntel`).** `analyzeExportSnapshot` computes a deterministic `apiIntel` block (endpoint payload coverage %, field→entity mapping coverage %, sensitive-exposure count, public-endpoint-risk count, capped supporting lists) by reshaping the export snapshot and calling the same analyzer primitives. It surfaces as a report-plan-gated **"API Payload Intelligence" PDF section** and as `apiIntel.*` keys in the **AI Review digest evidence allow-list** — the model may *cite* these metrics but never *compute* them (Safety Rule 3); `hashAnalysis` includes the block, so reviews correctly go stale when payloads change. (`SnapshotEndpoint`/`SnapshotDatabaseModel` were widened to declare the `requestSchema`/`responseSchema`/entity fields the runtime snapshot already carried — no data change.)

**Frontend (read-only).** `lib/api/api-intel.ts` wraps the endpoint; `components/api/` renders two lenses inside the expandable endpoint row on the API-spec detail page — **Impact Analysis** (default: Touches / Implemented By / Protected By / Referenced In / Workflow / Payload Fields / Warnings) and **Architecture Links** (chips + Workflow Impact) — both labelled `· Inferred ·`, confidence-dotted, with `basis` tooltips and chips linking to the real resource. The graph overlay consumes the same `inferredEdges`. DTO shape is stable, so the analyzer's heuristic refinements need no frontend change.

**Testbed.** `scripts/seed-payload-testbed.ts` (**`npm run seed:testbed`**) seeds a separate, idempotent **"Payload Intelligence Testbed"** project that exercises the whole chain end-to-end, including intentionally-bad endpoints (e.g. `POST /debug/leak-token`) that prove the validation rules fire. It find-or-creates the demo user and never touches the main demo.

### Export Engine V2 (SSOT export + PDF report)

Three layers, strictly separated — each consumes the previous and never reaches back:

```
buildExportContent(projectId, format, sections)   ← SSOT assembly (single source)
        ↓ persisted to ExportPackage.content (Json)
analyzeExportSnapshot(content) → AnalysisResult    ← deterministic analysis (pure)
        ↓
renderArchitecturePdf({content, analysis, meta})   ← PDF presentation (pure)
```

- **SSOT assembly** — `modules/exports/exports.engine.ts:buildExportContent` is the **only** place export data is assembled. JSON returns the payload object; MARKDOWN returns a string; PDF gets the same object. Do not duplicate this assembly anywhere; the analysis and PDF layers read its output, never the DB.
- **Analysis engine** — `modules/exports/analysis/` (`metrics.engine.ts` + `analysis.constants.ts` + `analysis.types.ts`). `analyzeExportSnapshot(content)` is a **pure, deterministic** function: no I/O, no Prisma, no `Date.now()` (the only time reference is `snapshot.generatedAt`), no AI. Same snapshot ⇒ deep-equal `AnalysisResult`. It computes health score (weighted composite), documentation coverage, connectivity, traceability, governance, validation, and rule-keyed risks. All weights/thresholds live in `analysis.constants.ts`. This is the foundation any future AI analysis must build on — keep it independent of the renderer.
- **PDF renderer** — `modules/exports/pdf/` on `pdfmake` (standard-14 fonts, no embedded TTF, **no headless browser**). `pdf.renderer.ts` is presentation-only: it reads numbers from `AnalysisResult`, never recomputes a score. Determinism is pinned by overriding pdfkit's `CreationDate`/`ModDate` and the file `_id` from the snapshot identity (otherwise every render differs). Missing logo → text fallback (`pdf/logo.ts`, the MINOTAURUS mark as inline SVG).
- **Report composition** — `pdf/report-plan.ts:buildReportPlan(sections, analysis, content)` is a pure, deterministic step that decides **which** PDF sections render from the selected `ExportPackage.sections` scope plus available data. The renderer gates every section on `plan.include`, so a `DIAGRAMS`-only export yields a short diagram report (cover + contents + Diagram Inventory + metadata) instead of ~18 empty architecture pages. Because unrendered sections never reach the document and section headers carry `tocItem`, the auto-TOC always matches what rendered — no phantom rows. Empty scope = full report (back-compat). The cover/title adapt via `plan.cover` / `plan.reportTitle` ("Architecture Diagram Report" vs "Architecture Intelligence Report"); diagrams are a **primary** section (`diagramsSection`), and each appendix subsection is gated to its scope. When adding a new section, add its flag here and gate it in `buildDocDefinition` — don't spread it unconditionally.
- **Download** — `GET /exports/:exportId/download` (`exports.controller.ts:downloadExport`, registered **before** the `:exportId` catch-all). PDF is rendered on demand from the persisted snapshot and streamed `application/pdf`; JSON/MARKDOWN stream the stored content. Create still requires `ARCHITECT`+; download mirrors read access.
- **`ExportFormat` is `JSON | MARKDOWN | PDF` — there is no `ZIP`.** ZIP was an enum value that was never implemented (its download silently fell through to JSON); it was removed from the Prisma enum (`EXPORT_FORMATS` and TS types must stay in sync) in migration `20260602120000_remove_zip_export_format` — 0 rows used it. The create endpoint validates `format` against `EXPORT_FORMATS`, so a `ZIP` request now returns `400 VALIDATION_ERROR`. Don't re-add `ZIP` as a placeholder; if a real bundle is built later, ship an actual multi-file archive (JSON + Markdown + PDF + diagram SVGs) with deterministic entry timestamps/order — mirror the PDF determinism pinning, never a JSON-in-disguise download.

**Diagrams in the PDF (the fiddly part — `pdf/diagram-svg.ts`).** Mermaid only renders in a browser, so the **frontend captures each diagram's SVG at export-create time** (`renderMermaidToSvg` in `mermaid-preview.tsx`, using `htmlLabels:false` so labels are native `<text>`) and POSTs a `diagramSvgs` map; the controller freezes it into the snapshot's `diagrams[].renderedSvg`. At render time `normalizeMermaidSvgForPdf` rewrites that SVG for `pdfmake`'s bundled `svg-to-pdfkit`, which has real limitations that drove a series of fixes — **understand these before touching that file**:

- **Sizing**: `SVGMeasure` reads the first non-percentage `width=` in the document; Mermaid's root is `width="100%"`, so it would match a child `<rect width=…>` and render the whole diagram compressed. Fix: rewrite the root `<svg>` with explicit numeric width/height from the viewBox.
- **`<foreignObject>` is silently dropped** → such SVG is rejected and falls back to the Mermaid source block (which is **always shown** beneath the rendered diagram regardless).
- **Contrast**: the frontend captures a dark UI theme; `svg-to-pdfkit` resolves CSS by specificity with no `!important`, so the `<style>` block is stripped and colors are rewritten at the string level to the print palette (`PRINT` constants).
- **`stroke-dasharray:"1,0"`** crashes pdfkit's `dash()` → invalid (non-positive) dasharrays are stripped.
- **Labels**: node/edge labels are recentred on their group origin (`text-anchor=middle`, `dominant-baseline=central`); their background `<rect>` is filled **white** (recentred to match) to mask connector lines; `<line>` lifelines get a visible stroke + 1pt min width.

Everything here is presentation-only — **never** change scoring, `AnalysisResult`, the SSOT assembly, or the JSON/Markdown outputs to fix a PDF rendering issue. `pdf/*.test.ts` and `analysis/*.test.ts` are the determinism contract; run `test:unit` after edits.

### Knowledge graph rendering (edges, layout, persistence)

[`components/graph/graph-canvas.tsx`](frontend/nextjs/components/graph/graph-canvas.tsx) is React Flow plus four custom layers; changes to any one need to stay aware of the others.

**1. Edges — `LabeledSmoothStepEdge` (custom edge type).** All edges render through this component instead of React Flow's built-in `smoothstep`:

- **Routing is smoothstep-first.** The clean orthogonal `getSmoothStepPath` is the default; the edge only escalates to the expensive `getSmartEdge` pathfinding (`@tisoap/react-flow-smart-edge`, **pinned to v3** — v4 is for `@xyflow/react` 12, incompatible with `reactflow` 11) when a cheap-path crossing test (`pathCrossesNode` of the reconstructed smoothstep waypoints vs every node box, inflated by `SMOOTHSTEP_NODE_PADDING`) shows the straight route would clip a node. This is *inverted* from the old smart-first default: it gives cleaner lines **and** runs pathfinding on only the few crossing edges. `getSmartEdge` is memoized on its routing inputs so unrelated re-renders don't re-pathfind.
- **During an active drag, all edges force the cheap `getSmoothStepPath`** (no `getSmartEdge`, no label work) so lines follow the moving node at 60fps; smart routing returns on drop. The drag flag rides on `data.dragging` (set from the `Inner`-level `draggingId`). Running smart routing live re-pathfinds every edge per frame → jank; don't.
- **Label placement + global de-collision.** Each edge measures its label anchor (`getPointAtLength(len/2)` on a duplicate invisible `<path>` — `getSmartEdge`'s `edgeCenterX/Y` is a routing *waypoint* and lands off-center) and **reports it to a shared `LabelLayoutContext` coordinator in `Inner`**, which runs a deterministic greedy pass (`resolveLabelLayout`: id-ordered, push overlapping boxes apart in y until clear) and hands each edge a final, guaranteed-non-overlapping position to render at. Two invariants keep this loop-free: (1) the reported anchor is the measured *path* point, **independent** of the label's resolved offset — so re-resolving never moves what's reported; (2) the measurement effect depends on the **stable `report` callback**, never the whole context value (whose `resolved` map changes every pass) — depending on the context object infinite-loops. Label state starts `null` so it never flashes pre-measurement.
- **Perf — labels & zoom.** Edges hide their labels below `LABEL_ZOOM_THRESHOLD` by subscribing to `useStore((s) => s.transform[2] >= THRESHOLD)` — a **boolean**, so edges re-render only when crossing the threshold, not on every zoom tick. Subscribing to the raw zoom number re-renders all edges per delta (which then re-pathfind and re-measure) → the graph stutters; keep it a boolean. The reflow-heavy `getTotalLength`/`getPointAtLength` measurement is also gated to "label actually visible" (not zoomed out, not mid-drag).
- **Label DOM**: rendered through `<EdgeLabelRenderer>` (a React Flow portal above the nodes layer) instead of the edge SVG (below nodes). A defensive `.react-flow__edgelabel-renderer { z-index: 10 }` in [`app/globals.css`](frontend/nextjs/app/globals.css) keeps it above nodes even if React Flow's internal DOM order changes.
- Edge color travels on `data.color`; the edge component reads it from there so the label border can tint to match the relation type.

**2. Layout — [`lib/graph-layout.ts`](frontend/nextjs/lib/graph-layout.ts).** Dagre LR/TB auto-layout, wired into `GraphCanvas` via two props that *both* defer to persisted drag positions per-node:

- `autoLayout="LR"|"TB"` — used by the artifact-detail mini-graph (a focused 1-hop subgraph) so neighbors line up cleanly.
- `relayoutSignal: number` — a parent-owned counter; incrementing it (from a toolbar button) wipes saved drag positions and re-runs dagre. The full project graph page's "Relayout" toolbar button drives this. After a relayout writes new positions the canvas also re-runs `fitView` (gated on a `pendingFitRef` so drag-persist and mount don't refit) via a **nested `requestAnimationFrame`** — a single frame fires before React Flow has committed the new positions and would frame the stale layout.
- `minZoom` prop (default `0.25`) — the floor `fitView` can zoom out to. Embedded mini-graphs (e.g. the dashboard, `minZoom={0.05}`) need a lower floor or `fitView` clamps and a large graph overflows the small container instead of fitting.
- **Layout reads the FULL real relation set, never the visible subset.** The full graph page's "Edges" dropdown can hide real edges, but `GraphCanvas` always receives the complete `relations` for dagre and uses a separate **`hideRealEdges`** prop to skip *rendering* them (edge visibility ≠ layout). Feeding dagre an emptied/visible-only relation set collapses the now-disconnected nodes into a vertical stack — so don't pass a filtered set as `relations` to control visibility; pass the full set + `hideRealEdges`. Inferred overlay edges (`inferredEdges`) likewise don't participate in layout.

**3. Position persistence + cross-view sync.** Drag positions persist to `localStorage` under `mino:graph:<storageKey>`. The **dashboard mini-graph and the full project graph share the same `storageKey` (`project:${projectId}`)**, so dragging in either view updates positions in the other on the next mount. Don't accidentally re-namespace one without the other. The artifact-detail subgraph has no `storageKey` by design (and the **public landing page no longer renders `GraphCanvas` at all** — its graph visuals are standalone SVG; see the node-style note below).

**Auto-relayout after AI Bootstrap.** AI-applied artifacts get random scatter `gx/gy` (`bootstrap.apply.ts:gxgy()`), so a freshly bootstrapped project would otherwise open as an overlapping mess. The Bootstrap Wizard sets a one-shot flag `mino:graph:relayout:<projectId>` in `localStorage` on apply; the **full graph page** (`graph/page.tsx`) consumes it once on load (after artifacts arrive) and bumps `relayoutSignal` to run the same dagre arrange the toolbar "Relayout" button drives, then clears the flag — so the first view is tidy but manual drags are respected on every later visit. This is a deliberate cross-file contract (wizard writes the flag, graph page consumes it); only the full graph page consumes it, not the dashboard mini-graph.

**4. Drag, selection, node style.** Non-trivial bits in `GraphCanvas`:

- **Live drag via controlled nodes**: nodes flow through `useNodesState`/`onNodesChange`, with `rfNodes` mirrored from the derived `derivedNodes` by an effect. The node follows the cursor because React Flow can apply per-frame deltas. The mirror effect must **not** fire mid-drag — none of `derivedNodes`' inputs change during a drag, so it doesn't; if you add a dep that changes during drag, the dragged node snaps back. (Earlier versions had no `onNodesChange` and the node only jumped on drop.)
- **Drop-time collision resolution** still runs only in `onNodeDragStop`: an iterative AABB push-out (≤16 passes) slides the dropped node out of overlaps, writing to `rfNodes` (the controlled source). **Don't** move this *collision* step into `onNodeDrag` — `setNodes` inside the live loop fights React Flow's drag delta and the node feels stuck. (Live *movement* is fine; live *collision* is not — they're independent.)
- **Last-dragged on top**: `lastDraggedId` assigns `zIndex: 1` to the most recently dropped node; React Flow already elevates the actively-dragged one, so the transition is seamless.
- **Focus mode**: when a node is selected (and `highlightSelected`), the canvas dims everything outside the selected node's 1-hop neighborhood (`neighborIds` from `relations`) — node opacity via `node.style`, edges via reduced `style.opacity`/emphasized `strokeWidth`. Pure render-time; no data change. `null` neighbor set = no focus, so the graph looks normal when nothing is selected.
- **`highlightSelected` prop** controls whether `selectedId` paints the accent border *and* whether focus mode runs. Off on the artifact-detail mini-graph (focal node implied by the route).
- **Node style is global to authenticated graphs.** All in-app graphs (full graph page, dashboard mini-graph, artifact-detail subgraph) pass `nodeStyle={graphNodeStyle}` from the `useTweaks` store, so the Settings → Workspace "Graph node style" preference drives every one live. The **public landing page does not use `GraphCanvas` or the auth store at all** — its graph visuals are hand-authored SVG in `components/landing/` (the decorative `parallax-graph.tsx` hero backdrop + the animated `pipeline-theater.tsx` knowledge-graph centerpiece). `pipeline-theater.tsx` deliberately **hardcodes its own type-color palette** (mirrored from `TYPE_INFO` in `lib/mock-data.ts` — keep the two in sync) and styles its SVG node cards to match the in-app `"color"` `MinoNode` (4px type band, fg title, type-colored uppercase label). This is the same public/auth separation as the accent FOUC split — a public page must not read `useTweaks`/`useAuth`.

## AI Safety & Determinism Rules

Minotaurus is a deterministic-first platform. AI is an **additive proposal-and-explanation layer that lives outside the deterministic core**: it may *read* the SSOT and `AnalysisResult` and *emit* structured proposals, but it re-enters the system only through the same human-gated, AI-free path ingestion uses — `propose → review → confirm → deterministic apply`. These rules are mandatory for every present and future AI feature and extend the determinism conventions below.

1. **AI never writes to the database directly.** AI code must not call `prisma.*.create / update / delete`. The only route from model output to persisted state is `AI → proposal → user review → user confirm → deterministic apply → DB`, where *apply* reuses the existing controllers/engines (the same creation logic ingestion-confirm calls) — never a parallel AI write path.
2. **AI output is not SSOT until a user confirms it.** Everything the model emits stays draft/proposed — never persisted, scored, validated, exported, or shown as real project state — until explicit user confirmation.
3. **`AnalysisResult` stays deterministic.** `AnalysisResult → AI` is allowed (AI may *explain* a score). `AI → AnalysisResult` is forbidden (AI may never *compute* or influence a score). The analysis engine stays pure (no AI, no `Date.now()`). If AI prose is ever embedded in an export, freeze it into the snapshot first — like diagram SVGs — so the PDF stays a pure function of stored bytes.
4. **Validation stays deterministic.** AI may *explain* validation findings and *suggest* fixes; AI may **not** create, modify, or resolve `ValidationIssue` rows. The rule-based validation engine remains the only writer of validation state.
5. **Every AI-proposed entity must pass deterministic validation before it joins the project.** Proposals are checked against the same invariants the controllers enforce — artifact title normalization + per-project uniqueness, in-range enums, relation endpoints that resolve, no duplicate relations, valid Mermaid, schema validity — *before* apply. Invalid items are rejected or flagged for the user, never silently applied.

**Provenance.** Every AI-applied change records a `VersionEvent` like any other mutation, with the confirming human as `triggeredBy` and origin metadata — e.g. `{ "origin": "AI", "source": "BOOTSTRAP_WIZARD", "confirmedBy": "<user-id>" }` — so the timeline never misrepresents what produced a change.

The first feature built on these rules is the **AI Bootstrap Wizard** (`modules/ai/`): from an existing empty project, the user describes an idea, AI proposes artifacts (created as `DRAFT`) + relations + one to three Mermaid diagrams, the user reviews and selects, and confirmed items are applied through the existing artifact/relation/diagram creation paths. A lightweight `AiSession` row (audit metadata, like `IngestionRecord`) records each propose/apply. AI does not create the project (it already exists) and does not write anything until confirm.

The **AI Architecture Review** is the first *read-only* feature on these rules: it reads the SSOT + deterministic `AnalysisResult` and emits a structured, evidence-verified narrative — never writes SSOT, never computes a score, no apply path. (`AnalysisResult → AI` allowed; `AI → AnalysisResult` forbidden — Rule 3.) It is **one surface with two modes** sharing the deterministic chain + generation loop (`modules/ai/architecture/`): **Full Review** (the audit, `modules/ai/review/`, persisted as `AiSession(REVIEW)`) and **Advisor** (a prioritized action plan, `modules/ai/advisor/`, persisted as `AiSession(ADVISOR)`). Both are read-only w.r.t. architecture — each may write **only its own** `AiSession` record (audit metadata / AI output, never SSOT or a score).

The **AI Documentation Assistant** (`modules/ai/documentation/`) is the third feature: per-artifact, on-demand drafting of an artifact's Markdown docs from a bounded artifact-only digest. AI returns a draft; it has **no save path** — the user reviews/edits and persists through the existing `PUT /artifacts/:id/documentation`. Existing docs are never overwritten by AI (improvement-suggestion mode). Audit metadata only (`AiSession(DOCUMENTATION_DRAFT)`).

## Conventions (from `docs/AI_CONTEXT/ARCHITECTURE_RULES.md`)

- Thin controllers — push non-trivial logic into `<feature>.engine.ts`.
- Frontend pages import typed wrappers from `lib/api/`; do not call `fetch` directly and do not duplicate DTO types (use `lib/types.ts`).
- Validation is rule-based and deterministic. Do not introduce AI-generated validation logic.
- Export Engine V2 is layered: SSOT assembly → analysis → PDF. The analysis engine and PDF renderer are pure and deterministic (no AI, no `Date.now()`). Don't recompute scores in the renderer or assemble SSOT data outside `buildExportContent`.
- `ArtifactRelation` is the graph source of truth — don't add a parallel "links" table or emit non-artifact nodes from the graph endpoint.
- Postgres is the source of truth (since Phase 6). The old `backend/src/db/data.json` JSON store is gone — don't reintroduce it.
- AI proposes, a deterministic apply disposes — `modules/ai/` never writes to the DB directly, and AI-generated Mermaid is structure-only (the renderer owns styling). See the AI Bootstrap Wizard section.
